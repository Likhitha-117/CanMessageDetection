MODULE 10: Model Deployment Preparation
--------------------------------------
1. Concept:
   - Converting a trained Deep Learning model (BiLSTM) into a deployable format and setting up a pipeline to load it for real-time inference.
2. What exactly I implemented:
   - Model Loading Utility (`model_loader.py`) to manage .h5/keras artifacts.
   - Scaler and Class Label loading using `joblib` and `numpy`.
   - Real-time sliding window buffer using `collections.deque`.
3. How it works step-by-step:
   - Startup: The FastAPI lifespan event calls `model_loader.load_all()`.
   - Artifacts: The system loads `bilstm.h5` (model), `lstm_minmax_scaler.pkl` (scaling), and `class_labels_lstm.npy` (mapping).
   - Preparation: A singleton `IntrusionDetector` class is initialized with a per-vehicle `deque` buffer of size 10.
4. Code explanation:
   - `tf.keras.models.load_model(str(MODEL_PATH))`: Loads the trained BiLSTM architecture and weights.
   - `joblib.load(str(SCALER_PATH))`: Loads the MinMax scaler used during training to ensure input features are in the same [0,1] range.
   - `deque(maxlen=settings.SEQ_LENGTH)`: Automatically maintains the last 10 CAN messages for each vehicle.
5. Real-time working explanation:
   - When the backend starts, it "warms up" by loading the model into memory. As CAN data arrives, it is scaled and pushed into a 10-step buffer. Once the buffer is full, the model processes the 10x10 matrix to predict if the sequence represents an attack.
6. Short viva explanation:
   - "I implemented a model loader that loads a BiLSTM model and a MinMax scaler. It uses a sliding window buffer of size 10 to feed sequences to the model for real-time intrusion detection."

MODULE 11: Backend Development
----------------------------
1. Concept:

2. What exactly I implemented:
   - FastAPI framework setup with asynchronous database (Motor/MongoDB) integration.
   - Modular API routing (Auth, Admin, Owner, Engineer, Logs, Analysis).
   - Log Ingestion and Batch Analysis endpoints.
3. How it works step-by-step:
   - Entry Point: `main.py` initializes the FastAPI app and registers routers.
   - Database: `core/database.py` connects to MongoDB Atlas using `AsyncIOMotorClient`.
   - Validation: Pydantic schemas in `models/schemas.py` enforce strict data types for every API request.
4. Code explanation:
   - `@app.post("/api/logs/ingest")`: Main endpoint for real-time data from vehicles.

   - `Depends(get_vehicle_by_api_key)`: Custom security dependency that validates vehicle identities before ingestion.
5. Real-time working explanation:
   - Vehicles send JSON payloads containing CAN ID, DLC, and Data bytes. The backend validates the data, runs the BiLSTM prediction via the `IntrusionDetector` service, and stores the results in MongoDB.
6. Short viva explanation:
   - "I used FastAPI for the backend because of its native async support. It handles real-time log ingestion, processes predictions, and stores all data in MongoDB Atlas."

MODULE 12: Real-Time Simulation
-----------------------------
1. Concept:
   - Emulating continuous vehicle traffic by streaming realistic CAN bus data from pre-recorded datasets to the backend.
2. What exactly I implemented:
   - A Python-based `vehicle_simulator.py`.
   - CSV-to-JSON streaming logic using `requests`.
   - Randomized delay emulation (50ms - 100ms) to mimic real CAN bus frequency.
3. How it works step-by-step:
   - Initialization: Simulator takes a `VEHICLE_API_KEY` and a CSV file path.
   - Reading: Iterates through the CSV rows using `csv.DictReader`.
   - Transformation: Combines `data_0` through `data_7` into a single `data` list.
   - Streaming: Sends a POST request to `/api/logs/ingest` with the CAN payload.
4. Code explanation:
   - `can_data = [int(row[f"data_{i}"]) for i in range(8)]`: Reconstructs the 8-byte CAN payload from CSV columns.
   - `requests.post(API_URL, json=payload, headers=headers)`: Transmits the data to the IDS backend.
   - `time.sleep(random.uniform(DELAY_MIN, DELAY_MAX))`: Simulates real-world message timing.
5. Real-time working explanation:
   - Once started, the simulator acts as a "Virtual Vehicle". It reads row-by-row, sending data to the server and waiting for a response that confirms whether the last sequence was "Normal" or an "Attack".
6. Short viva explanation:
   - "I created a vehicle simulator that streams CAN data from CSV files to the API. It mimics a real ECU by sending data packets with randomized delays and handles the ingestion headers for security."

MODULE 14: Authentication & Security
----------------------------------
1. Concept:
   - Securing the platform by ensuring only authorized users can access data, using modern token-based authentication.
2. What exactly I implemented:
   - JWT (JSON Web Token) authentication system.
   - Password hashing with `bcrypt` (`passlib`).
   - Token-based API security via `PyJWT`.
3. How it works step-by-step:
   - Register: User details and hashed passwords are saved in MongoDB.
   - Login: Backend verifies the password and returns a JWT containing the user's `id` and `role`.
   - Storage: Frontend stores the token in `localStorage`.
   - Access: Submitting the token in the `Authorization` header for every protected request.
4. Code explanation:
   - `pwd_context.hash(password)`: Salting and hashing passwords before storage.
   - `jwt.encode(to_encode, settings.SECRET_KEY)`: Generates a cryptographically signed token.
   - `require_role("admin")`: A FastAPI dependency that extracts the role from the JWT and blocks unauthorized access.
5. Real-time working explanation:
   - If a user tries to access the dash without a token, the `api.js` interceptor detects the missing session and redirects them to the login page. Tokens have an expiration time (e.g., 60 minutes) to minimize risk.
6. Short viva explanation:
   - "I implemented JWT-based authentication. Passwords are encrypted using bcrypt, and every API request is protected by a token validation middleware that checks both identity and user roles."

MODULE 15: Role-Based Access Control (RBAC)
-----------------------------------------
1. Concept:
   - Restricting functionality and data visibility based on user roles to satisfy security requirements.
2. What exactly I implemented:
   - Roles: `admin`, `owner`, and `engineer`.
   - Payload Restriction: Only `engineer` can see the raw 8-byte CAN payload.
   - Admin/Owner restricted views (Summary only).
3. How it works step-by-step:
   - Assignment: Admins create vehicles and assign them to specific Engineers.
   - Validation: The `require_role` dependency checks if the current user has the authority to call an endpoint.
   - Data Masking: The `logs` API masks or removes the `payload` field if the user is not an Engineer.
4. Code explanation:
   - `user=Depends(require_role("engineer"))`: Used in `api/engineer.py` to ensure only experts access forensic tools.
   - `v.pop("vehicle_api_key", None)`: Ensures sensitive keys aren't returned to unauthorized users.
5. Real-time working explanation:
   - If an `owner` views logs, they see the "Timestamp", "CAN ID", and "Prediction". If an `engineer` views the same logs, they see an extra column for "Raw Payload" (Data Bytes), allowing for deeper technical investigation.
6. Short viva explanation:
   - "My system defines three roles. I implemented backend logic to ensure sensitive CAN payloads are only visible to Security Engineers, while Admins and Owners see higher-level summaries."
